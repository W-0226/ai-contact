## 二、测试用例

### TC01 — 老王（信息完整型）

**输入数据：**

```
老王，3月15号生日，属兔，喜欢钓鱼和打乒乓球。
上次7月初聊到他在做跨境电商，主要做东南亚市场。
他老婆叫小李，孩子刚上小学。
```

**测试目的：** 验证本地规则引擎对信息完整联系人的提取准确性。

**执行日志：**

```
========== TC01_老王 ==========
姓名: 老王
生日: {'month': 3, 'day': 15, 'type': 'solar', 'raw': '3月15号'}
爱好: ['钓鱼', '打乒乓球']
事件: [{'date': '7月', 'description': '聊到他在做跨境电商'}]
标签: ['已婚', '有孩子']
置信度: 0.76
价值评分: 61分 (medium)
  信息:20 亲密:16 联系:20 潜力:5
断联状态: 活跃
待确认项: 1 项
  [medium] 断联原因: 「老王」已较长时间未联系，需要判断原因...
关系分析章节: 已生成
```

**逐字段验证：**

| 字段 | 期望值 | 实际值 | 结果 |
|------|--------|--------|:--:|
| 姓名 | 老王 | 老王 | ✅ |
| 生日 | 3月15日（公历） | `{month:3, day:15, type:"solar"}` | ✅ |
| 爱好 | 钓鱼、乒乓球 | `["钓鱼", "打乒乓球"]` | ✅ |
| 关键事件 | 7月聊跨境电商 | `{date:"7月", desc:"聊到他在做跨境电商"}` | ✅ |
| 标签 | 已婚、有孩子 | `["已婚", "有孩子"]` | ✅ |
| 置信度 | ≥0.70 | 0.76 | ✅ |
| 价值评分 | 中等 | 61分（medium） | ✅ |

**JSON 输出：**

```json
{
  "name": "老王",
  "birthday": {"month": 3, "day": 15, "type": "solar", "raw": "3月15号"},
  "hobbies": ["钓鱼", "打乒乓球"],
  "key_events": [{"date": "7月", "description": "聊到他在做跨境电商"}],
  "tags": ["已婚", "有孩子"],
  "confidence": 0.76,
  "source": "local"
}
```

**已知瑕疵（不影响核心功能）：**
- 生肖「属兔」未被提取（规则未覆盖）
- "7月初"被解析为7月15日附近（距今天数偏小）

## 五、测试结论

### 通过的验证点

- ✅ **核心提取**：姓名、生日（公历/农历/模糊）、爱好、标签、关键事件均能正确提取
- ✅ **诚实性**：信息缺失时返回 null/[]，不编造；置信度准确反映信息量
- ✅ **关系分析**：四维价值评分合理区分高/中/低价值联系人；断联状态分级准确
- ✅ **确认队列**：自动检测 6 类模糊关系，每项提供 2-5 个可选方案
- ✅ **隐私保护**：AES-256 加密/解密正常，密码错误正确拒绝，分享令牌可用
- ✅ **稳定性**：90 个单元测试全部通过，None 值安全处理，空输入不崩溃
- ✅ **全流程**：从输入到输出（JSON + Markdown + 加密 + 确认队列）端到端正常

### 待改进项

- ⚠️ 姓名提取在复杂分段场景下准确率待提升
- ⚠️ 事件日期与生日日期的上下文区分需加强
- ⚠️ 标签词典需扩展（CTO、VP 等职位头衔）
- ⚠️ hybrid 模式（LLM 增强）需 API key 方可验证
- ⚠️ OCR/ASR 模块需安装对应依赖后进行集成测试

### 总体评价

**本轮测试通过。** 核心功能（提取、评分、预警、确认、加密）均正常运作，90 个单元测试全部通过。两个已知 bug（B-01/B-02）影响有限，不影响主要流程。建议在提交前修复 B-02（姓名误识别）。
